# Alembic Migrations with Neon PostgreSQL

This guide explains how to set up and run database migrations with Alembic for your Neon PostgreSQL database.

## Prerequisites

1. ✅ Neon database connection string configured in `backend/.env`
2. ✅ `psycopg2-binary` installed
3. ✅ Virtual environment activated

## Initial Setup

### Step 1: Ensure Your .env File is Configured

Make sure your `backend/.env` file has your Neon connection string:

```env
DATABASE_URL=postgresql://user:password@ep-xxx-xxx.us-east-2.aws.neon.tech/neondb?sslmode=require
```

### Step 2: Create Initial Migration

Generate your first migration based on your models:

```powershell
cd backend
.\venv\Scripts\Activate.ps1
alembic revision --autogenerate -m "Initial migration"
```

This will:
- Compare your SQLAlchemy models with the current database state
- Generate a migration file in `alembic/versions/`
- Create the necessary `upgrade()` and `downgrade()` functions

### Step 3: Review the Migration File

Check the generated migration file in `alembic/versions/`:

```powershell
# Open the latest migration file to review
code alembic/versions/xxxx_initial_migration.py
```

Make sure it includes:
- Creating `users` table
- Creating `checkinouts` table
- All necessary columns and constraints

### Step 4: Apply the Migration

Run the migration to create tables in your Neon database:

```powershell
alembic upgrade head
```

This will:
- Connect to your Neon PostgreSQL database
- Create all tables defined in your models
- Track the migration in the `alembic_version` table

## Common Migration Commands

### Create a New Migration

```powershell
# Auto-generate migration from model changes
alembic revision --autogenerate -m "Description of changes"

# Create empty migration (manual)
alembic revision -m "Description of changes"
```

### Apply Migrations

```powershell
# Apply all pending migrations
alembic upgrade head

# Apply migrations up to a specific revision
alembic upgrade <revision_id>

# Apply next migration only
alembic upgrade +1
```

### Rollback Migrations

```powershell
# Rollback one migration
alembic downgrade -1

# Rollback to a specific revision
alembic downgrade <revision_id>

# Rollback all migrations
alembic downgrade base
```

### Check Migration Status

```powershell
# Show current revision
alembic current

# Show migration history
alembic history

# Show pending migrations
alembic heads
```

## Example Workflow

### Scenario: Adding a New Field to User Model

1. **Update your model** (`app/models/user.py`):
```python
class User(Base):
    # ... existing fields ...
    phone_number = Column(String, nullable=True)  # New field
```

2. **Generate migration**:
```powershell
alembic revision --autogenerate -m "Add phone_number to users"
```

3. **Review the migration file**:
```powershell
# Check alembic/versions/xxxx_add_phone_number_to_users.py
```

4. **Apply migration**:
```powershell
alembic upgrade head
```

## Troubleshooting

### Migration Fails with "Table Already Exists"

If you've already created tables manually or via `Base.metadata.create_all()`:

**Option 1: Stamp the database (recommended)**
```powershell
# Mark current database state as the initial migration
alembic stamp head
```

**Option 2: Drop and recreate (development only)**
```powershell
# WARNING: This deletes all data!
alembic downgrade base
alembic upgrade head
```

### Connection Errors

Make sure:
- Your `.env` file has the correct `DATABASE_URL`
- The connection string includes `?sslmode=require`
- `psycopg2-binary` is installed

### Migration Not Detecting Changes

If autogenerate doesn't detect model changes:
1. Make sure all models are imported in `alembic/env.py`
2. Check that models inherit from `Base`
3. Try creating a manual migration instead

### Multiple Heads Error

If you see "Multiple heads detected":
```powershell
# Merge the heads
alembic merge heads -m "Merge multiple heads"
alembic upgrade head
```

## Best Practices

1. **Always review autogenerated migrations** before applying
2. **Test migrations** on a development database first
3. **Use descriptive migration messages**: `-m "Add email verification field"`
4. **Commit migration files** to version control
5. **Never edit applied migrations** - create new ones instead
6. **Backup your database** before running migrations in production

## Migration File Structure

A typical migration file looks like:

```python
"""Initial migration

Revision ID: abc123
Revises: 
Create Date: 2024-01-01 12:00:00

"""
from alembic import op
import sqlalchemy as sa

# revision identifiers
revision = 'abc123'
down_revision = None
branch_labels = None
depends_on = None

def upgrade():
    # Create users table
    op.create_table('users',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('email', sa.String(), nullable=False),
        # ... more columns
        sa.PrimaryKeyConstraint('id')
    )
    
    # Create checkinouts table
    op.create_table('checkinouts',
        # ... table definition
    )

def downgrade():
    # Drop tables in reverse order
    op.drop_table('checkinouts')
    op.drop_table('users')
```

## Production Deployment

When deploying to production:

1. **Run migrations as part of deployment**:
```bash
alembic upgrade head
```

2. **Set up migration tracking** in your CI/CD pipeline

3. **Always backup** before running migrations

4. **Test migrations** on staging first
